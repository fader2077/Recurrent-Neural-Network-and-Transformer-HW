\subsection{Why the Original Corpus Failed}

The initial corpus was built from only 500 Simple Wikipedia pages selected by broad keyword matching. That caused two problems:
\begin{itemize}
    \item many pages were only loosely related to the final questions,
    \item even good re-ranking could not help much when the candidate pool was already weak.
\end{itemize}

\subsection{Corrected Retrieval Corpus}

The improved pipeline does not rely only on that static 500-page subset. Instead, it builds a targeted Wikipedia evidence set per question by:
\begin{enumerate}
    \item rewriting the question into a concise search query,
    \item fetching top Wikipedia pages with the MediaWiki API,
    \item downloading longer extracts,
    \item chunking the resulting pages,
    \item applying dense retrieval + re-ranking over those chunks.
\end{enumerate}

This correction is what unlocked the later re-ranking gains.

\section{Part 2: Retrieval and Re-ranking}

\subsection{Why the Old Hit Rate Was Misleading}

The first report used a coarse keyword-overlap hit rate. That metric only checked whether any keyword from the correct answer appeared somewhere in the retrieved top-3 chunks. It could not measure whether the best chunk moved upward after re-ranking.

That is why the first notebook showed identical scores for vector-only and re-ranked retrieval even though the re-ranker was changing the candidate ordering.

\subsection{Corrected Retrieval Experiment}

On the corrected science dev slice, a stronger downstream metric was used: final multiple-choice answer accuracy after retrieval.

\begin{table}[H]
\centering
\begin{tabular}{lr}

This is the key corrected finding for HW3:

\begin{quote}
\textbf{Re-ranking doubled answer accuracy from 30\% to 60\% on the corrected science dev slice.}

That directly addresses the original concern that re-ranking should be better. Under a stronger corpus and a stronger evaluation metric, it is better.

\subsection{Representative Re-ranking Examples}

\textbf{Example 1: MOND / galaxy clusters.} The query-rewritten retrieval found the correct concept page for Modified Newtonian dynamics. Vector-only retrieval still mixed in less useful context, while the re-ranker promoted the chunk describing the discrepancy reduction from roughly 10 to roughly 2, enabling the correct answer.

\textbf{Example 2: Roche limit.} The question-only retrieval found the correct page, but the best answer required the most precise chunk. After re-ranking, the chunk explicitly describing the tidal-disruption distance was moved to the top and the final system selected the correct option.

\subsection{Final Ensemble}

The strongest system combined three signals:
\begin{enumerate}
    \item \textbf{Option-aware evidence scorer} using query rewriting, long Wikipedia extracts, chunk-level re-ranking, and NLI-style scoring.
    \item \textbf{Question-context DeepSeek RAG} retrieving Wikipedia context for the question, re-ranking the chunks, then answering with DeepSeek.
    \item \textbf{DeepSeek model-only vote} used only when the faster evidence scorer is uncertain and the first two systems disagree.
\end{enumerate}

Final decision rule:
\begin{itemize}
    \item trust the option-aware scorer when its confidence gap is large,
    \item otherwise compare it with the question-context RAG answer,
    \item if needed, add the model-only vote,
    \item if all three disagree, ask a final DeepSeek judge over retrieved context.
\end{itemize}

In the corrected advanced pipeline, the dominant new cost comes from:
\begin{itemize}

    \item chunk-level re-ranking over longer extracts,
    \item DeepSeek generation on harder questions.
\end{itemize}

So the advanced system is substantially slower, but the accuracy gain is also substantial:
\begin{itemize}
    \item 20\% $\rightarrow$ 72\% held-out,
    \item 30\% $\rightarrow$ 60\% on the corrected DeepSeek RAG ablation when re-ranking is enabled.
\end{itemize}

This makes the engineering tradeoff reasonable: higher latency buys a much stronger answer quality signal.
